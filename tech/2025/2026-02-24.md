### Release Readiness

#### Alignment with “Core Ontology”

Discussion on whether SPDX Core should adopt existing ontologies for common elements (organization, person, location, etc.).

| Ontology Source | Description |
| --- | --- |
| **Common Core Ontologies (CCO)** | Extension of Basic Formal Ontology (BFO). Modules include Geospatial, Event, Time, Agent, Artifact, etc. |
| **Core Vocabularies (SEMIC)** | Reuse/extend W3C standards. Includes Business, Location, Person, and Public Organisation. |

> **Alignment Strategy:** SPDX Core already defines some of these concepts. We can align using `owl:sameAs` or create machine-readable mappings using [SSSOM](https://mapping-commons.github.io/sssom/).

### Notes

#### Signing Presentation - Jason Smith

* **Resource:** [SBOM Signing Best Practices](https://github.com/shiftleftcyber/sbom-signing-best-practices)
* **Current Gaps:** SPDX lacks a defined data model for signatures.
* **Binary Fragility:** Issues arise when lexical formatting is signed instead of identity.
* **Key Standards discussed:**
* RFC 8785: JSON Canonicalization Scheme (JCS)
* JSON Signature Format (JSF)
* ITU X.590: JSON Signature Scheme (JSS)


* **Discussion:** Compared SPDX "canonical serialization" to JCS. JSON-LD uses "sets" (unordered) rather than "lists." Issues will be created to further this discussion.

#### TEA Update

* **Project:** [Transparency Exchange API](https://github.com/CycloneDX/transparency-exchange-api)
* **Status:** Currently OpenAPI; moving toward [ECMA TC54](https://tc54.org/tea/) standardization.
* **Goal:** Automate discovery and download of SBOMs.
* **Timeline:** Community release (with CycloneDX 2.0) around Summer 2026; Official spec by end-of-year.
* **Next Steps:** Schedule a meeting to map SPDX terminology to TEA.
