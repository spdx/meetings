# Alignment with Core Ontologies

Discussion on whether **SPDX Core** should adopt elements from external ontologies.

These ontologies define common entities such as:

- organization
- person
- legal entity
- location
- event
- evidence

---

## Common Core Ontologies (CCO)

Extension of **Basic Formal Ontology (BFO / ISO/IEC 21838-2)**

Current modules include:

- Geospatial
- Information Entity
- Event
- Time
- Agent
- Quantity
- Units of Measure
- Currency Unit
- Facility
- Artifact
- Extended Relations

References:

- https://commoncoreontology.github.io/cco-webpage/  
- https://www.ontologyrepository.com/  
- https://github.com/CommonCoreOntology/CommonCoreOntologies  
- https://arxiv.org/abs/2404.17758

---

## Core Vocabularies (SEMIC / W3C)

Reuse and extend existing W3C standards.

Current vocabularies:

- Business (legal entity)
- Location
- Person
- Public Service
- Criterion & Evidence
- Public Organisation
- Assessment
- Public Event
- Standards & Specifications

References:

- https://interoperable-europe.ec.europa.eu/collection/semic-support-centre/core-vocabularies  
- https://mapping.semic.eu/  
- https://github.com/SEMICeu

---

## Alignment Considerations

SPDX Core profile already defines some concepts that overlap with **CCO** and **SEMIC Core**.

Possible alignment approaches:

- Use `owl:sameAs`
- Create machine-readable mappings using **SSSOM**

Reference:
- https://mapping-commons.github.io/sssom/

- https://github.com/bfo-ontology/BFO-2020

## model.ttl Invalid specVersion Issue
- https://github.com/spdx/spdx-spec/issues/1366

Possible solutions discussed:

1. Relax constraint for `specVersion` to allow format `x.y`
2. Relax constraint in SemVer definition
3. Explicitly state that if patch number is missing, **latest stable version is assumed**
